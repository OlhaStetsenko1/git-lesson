### 4.3.2 Defect Lifecycle

The defect lifecycle outlines the process and stages that a defect goes through from identification to resolution. Below is the detailed description of each stage in the defect lifecycle:

| Status       | Description                                                                 | Actions Required                                                                                           |
|--------------|-----------------------------------------------------------------------------|------------------------------------------------------------------------------------------------------------|
| New          | A defect is identified and logged for the first time.                       | The defect is logged in the defect tracking system with all necessary details and assigned to a developer. |
| Assigned     | The defect is assigned to a developer for analysis and resolution.          | The developer reviews the defect, analyzes the root cause, and plans the resolution.                       |
| In Progress  | The developer is actively working on fixing the defect.                     | The developer implements the fix and performs unit testing.                                                |
| Fixed        | The defect has been fixed by the developer.                                 | The developer marks the defect as fixed and assigns it to the tester for verification.                     |
| Retest       | The tester verifies the fix in the testing environment.                     | The tester retests the defect to ensure it has been resolved.                                              |
| Closed       | The defect has been successfully verified and closed.                       | The tester closes the defect if the fix is verified successfully.                                          |
| Reopened     | The defect is found to be not fixed or reoccurs after being closed.         | The tester reopens the defect and assigns it back to the developer for further analysis and resolution.    |
| Deferred     | The defect is postponed for fixing in a future release.                     | The defect is marked as deferred with a justification and planned for a future release.                    |
| Rejected     | The defect is found to be invalid or not reproducible.                      | The defect is marked as rejected with a justification.                                                     |

#### Rules for Each Transition/Status Change

1. **New to Assigned**: The defect is reviewed by the test lead or project manager and assigned to an appropriate developer.
2. **Assigned to In Progress**: The developer acknowledges the assignment and starts working on the defect.
3. **In Progress to Fixed**: The developer completes the fix and performs unit testing to ensure the defect is resolved.
4. **Fixed to Retest**: The defect is assigned back to the tester for verification.
5. **Retest to Closed**: The tester verifies the fix and closes the defect if it is resolved.
6. **Retest to Reopened**: If the defect is not resolved, the tester reopens it and assigns it back to the developer.
7. **Any Status to Deferred**: The defect is deferred with a valid reason and planned for a future release.
8. **Any Status to Rejected**: The defect is rejected with a valid reason if it is found to be invalid or not reproducible.

#### Available Resolutions

- **Fixed**: The defect has been resolved by the developer.
- **Deferred**: The defect is postponed for a future release.
- **Rejected**: The defect is invalid or not reproducible.
