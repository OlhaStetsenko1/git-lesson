### 4.3.1 Defect Reports Raising and Management Rules

| Field                | Description                                                                                       |
|----------------------|---------------------------------------------------------------------------------------------------|
| Defect ID            | Unique identifier for the defect.                                                                 |
| Summary              | A brief summary of the defect.                                                                    |
| Description          | Detailed description of the defect, including steps to reproduce, expected result, and actual result. |
| Severity             | The severity level of the defect (Blocker, Critical, Major, Minor, Trivial).                      |
| Priority             | The priority level of the defect (High, Medium, Low).                                             |
| Status               | Current status of the defect (New, In Progress, Fixed, Closed, Rejected, etc.).                   |
| Assignee             | The person responsible for fixing the defect.                                                     |
| Reporter             | The person who reported the defect.                                                               |
| Environment          | The environment in which the defect was found (e.g., OS, browser, device).                        |
| Attachments          | Any relevant screenshots, logs, or files that help in understanding or reproducing the defect.    |
| Comments             | Any additional comments or updates related to the defect.                                         |

#### Rules for Raising Defects:
1. **Defect Identification**: Ensure the defect is reproducible and not a duplicate of an existing defect.
2. **Severity and Priority Assignment**: Assign appropriate severity and priority based on the impact on the business/consumer.
3. **Detailed Description**: Provide a detailed description with clear steps to reproduce, expected and actual results.
4. **Environment Specification**: Specify the environment details where the defect was found.
5. **Attachments**: Include relevant attachments such as screenshots, logs, or any other files that can help in understanding or reproducing the defect.
6. **Assignee**: Assign the defect to the appropriate person or team responsible for fixing it.
7. **Status Updates**: Regularly update the status of the defect as it progresses through its lifecycle.

#### Defect Lifecycle:
1. **New**: The defect is newly reported and needs to be triaged.
2. **In Progress**: The defect is being actively worked on by the assignee.
3. **Fixed**: The defect has been fixed and is ready for retesting.
4. **Closed**: The defect has been verified as fixed and is closed.
5. **Rejected**: The defect is not considered valid or reproducible and is rejected.

#### Resolution Rules:
- **Fixed**: The defect has been resolved and the fix is verified.
- **Won't Fix**: The defect will not be fixed due to various reasons (e.g., low impact, out of scope).
- **Duplicate**: The defect is a duplicate of an existing defect.
- **Cannot Reproduce**: The defect cannot be reproduced based on the provided information.
