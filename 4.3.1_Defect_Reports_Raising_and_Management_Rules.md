### 4.3.1 Defect Reports Raising and Management Rules

| Field                | Description                                                                                       |
|----------------------|---------------------------------------------------------------------------------------------------|
| **Defect ID**        | Unique identifier for the defect.                                                                 |
| **Summary**          | A brief summary of the defect.                                                                    |
| **Description**      | Detailed description of the defect, including steps to reproduce, expected result, and actual result. |
| **Severity**         | The severity of the defect (Blocker, Critical, Major, Minor, Trivial).                            |
| **Priority**         | The priority of the defect (High, Medium, Low).                                                   |
| **Status**           | Current status of the defect (New, Assigned, In Progress, Fixed, Closed, Reopened).               |
| **Assigned To**      | The person responsible for fixing the defect.                                                     |
| **Reported By**      | The person who reported the defect.                                                               |
| **Date Reported**    | The date when the defect was reported.                                                            |
| **Environment**      | The environment in which the defect was found (e.g., OS, browser, device).                        |
| **Attachments**      | Any relevant attachments, such as screenshots or logs.                                            |

#### Rules for Raising Defects:
1. **Defect Submission**: All defects must be submitted through the designated defect tracking tool.
2. **Complete Information**: Ensure all mandatory fields are filled out with accurate and detailed information.
3. **Reproducibility**: Defects should be reproducible. If not, provide as much information as possible to help reproduce the issue.
4. **Severity and Priority**: Assign appropriate severity and priority based on the impact on the application and business.
5. **Attachments**: Include relevant attachments such as screenshots, logs, or videos to provide additional context.

#### Defect Lifecycle:
1. **New**: Defect is reported and awaits triage.
2. **Assigned**: Defect is assigned to a developer for fixing.
3. **In Progress**: Developer is actively working on fixing the defect.
4. **Fixed**: Developer has fixed the defect and it is ready for retesting.
5. **Closed**: Defect has been verified as fixed and is closed.
6. **Reopened**: Defect is reopened if it is found to be not fixed during retesting.

#### Review Process:
1. **Triage Meeting**: Regular triage meetings will be held to review new defects, assign severity and priority, and allocate them to developers.
2. **Verification**: Testers will verify the fixed defects and close them if the issue is resolved.
3. **Reopening**: If a defect is found to be not fixed, it will be reopened and assigned back to the developer.
